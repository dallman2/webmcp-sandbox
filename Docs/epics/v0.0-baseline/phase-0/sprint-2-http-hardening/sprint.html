Sprint 2: HTTP-Only Hardening

Phase: 0 Sprint: 2 Depends on: Sprint 1

Objective

Close the three architectural gaps identified after Sprint 1: (a) drop stdio transport and switch to rmcp's built-in StreamableHttpService for HTTP-only SSE; (b) eliminate the polyfill registration deadlock by injecting the polyfill into the page's MAIN world at document_start — guaranteeing document.modelContext exists before any page script runs; (c) fix multi-param tool support by wrapping tool inputs in Parameters<T> so the #[tool_router] macro generates correct CallToolHandler impls. Also add WebSocket keep-alive (server-initiated ping/pong to prevent MV3 service worker 30s idle disconnect), Zod validation on extension tool arguments, and resolve the remaining Sprint 1 formatting/cleanup drift.

Rationale & Context

Sprint 1 proved the full relay loop works but left three root-cause issues unaddressed: (1) the polyfill installs document.modelContext only in the realm where it evaluates — our content script ran in MV3's isolated world, so the page's main world never saw it (tools returned empty/null regardless of timing — not a race, a deadlock); (2) bare tool input structs without Parameters<T> wrapper failed IntoToolRoute because the macro fingerprint-matches the literal type name and FromContextPart has no blanket impl; (3) the stdio transport path was an empty stub and the manual JSON-RPC string dispatch in message_handler (~100 lines) was brittle. This sprint replaces each system with its correct rmcp canon. Additionally: the MV3 service worker disconnects after ~30s idle because no keep-alive traffic flows on the raw WebSocket (the @mcp-b/transports keep-alive only covers chrome.runtime.Port messaging, not raw WS); AGENTS.md:65 mandates Zod validation which was not implemented; and Sprint 1 left behind formatting drift and a dead-field clippy warning.

Scope

In-Scope

  • Drop transport-io from Cargo.toml; delete run_stdio and the --transport clap arg
  • Switch SSE from hand-rolled axum handler to rmcp's StreamableHttpService (feature transport-streamable-http-server) — eliminates the manual message_handler JSON-RPC string dispatch
  • Wrap tool inputs in Parameters<T>; use #[tool_router(server_handler)] to auto-emit the impl ServerHandler block
  • Delete manual ToolDef struct and tool_definitions() schema plumbing — rmcp's router generates schema from Parameters<T>
  • Inject the polyfill into the page's MAIN world via manifest static content_scripts with world: "MAIN" + run_at: "document_start"
  • Switch chrome.scripting.executeScript calls in background.ts to world: "MAIN" so injected functions share the polyfill's StrictWebMCPContext

  • Zod validation on background.ts tool arguments before forwarding to document.modelContext.executeTool
  • Run cargo fmt to clear formatting drift; drop dead ExtensionRequest.id field
  • Atomic vertical slice: group 1–6 land together in one Docker build cycle; all 13 verification gates must pass before any task is checked off

Execution Plan

1. HTTP-Only Transport

  1. Task 1.1: Drop transport-io, run_stdio, and --transport clap arg

    Remove "transport-io" from rmcp features in Cargo.toml. Delete the --transport field from Args and the run_stdio function from main.rs. SSE is the only transport.

  2. Task 1.2: Switch SSE to StreamableHttpService

    Change rmcp features to ["transport-streamable-http-server"] (transitively pulls server, macros, schemars, server-side-http). Wire StreamableHttpService::new(service_factory, session_manager, config) to an axum Router on :3000. This eliminates the entire manual message_handler JSON-RPC string dispatch (~100 lines deleted). After this, tools/list and tools/call are handled by rmcp's built-in ServerHandler::handle_request — we never touch raw "initialize"/"tools/list"/"tools/call" string matching again.

2. MAIN-World Polyfill Injection

  1. Task 2.1: Declare content script with world:MAIN + run_at:document_start

    Add world: "MAIN" and run_at: "document_start" to defineContentScript. Chrome runs MAIN-world content_scripts at document_start BEFORE the page's first <script>. Combined with the polyfill's synchronous init (index.d.ts:53: void return, no dynamic imports), document.modelContext is guaranteed present in main world before any page code executes. The fixture page no longer needs a polling loop.

  2. Task 2.2: Switch executeScript calls to world:MAIN

    Add world: "MAIN" to both chrome.scripting.executeScript calls in background.ts (for callPageTool and discoverPageTools). Without this, injected functions run in the default ISOLATED world where the polyfill is absent — they'd never see the registered tools. Post-fix, both the polyfill and the injected query functions share a single StrictWebMCPContext with a single tool Map.

  3. Task 2.3: Normalize discoverPageTools response for WS serialization

    Strip non-serializable fields from the discoverPageTools response: the polyfill returns {name, description, inputSchema, title, origin, window} where window is a live Window object that won't cross the WebSocket boundary, and inputSchema is a JSON string. Parse inputSchema back to an object and strip the window/title/origin fields before forwarding over WS.

3. Multi-Param rmcp Tools

  1. Task 3.1: Wrap tool inputs in Parameters<T>

    Import rmcp::handler::server::wrapper::Parameters. Change tool signatures to async fn execute_tool(&self, input: Parameters<ExecuteToolInput>) -> Result<model::CallToolResult, rmcp::ErrorData>. Access inner via input.0. Parameters<P: DeserializeOwned> implements FromContextPart which deserializes the MCP arguments JSON into P automatically — we delete the manual serde_json::from_value plumbing. The macro fingerprint-matches the literal type name to generate correct CallToolHandler impls.

  2. Task 3.2: Use #[tool_router(server_handler)] consolidation

    Replace the separate #[rmcp::tool_router] + #[rmcp::tool_handler] impl ServerHandler for McpHandler {} with #[rmcp::tool_router(server_handler)]. The server_handler flag auto-emits the impl ServerHandler block per MAC/src/tool_router.rs:90-95. Also delete the ToolDef struct and tool_definitions() plumbing — the router's list_all() handles tool listing from the registered #[tool] functions.

4. WS Keep-Alive

  1. Task 4.1: Server-side 20s ping interval over WS

    In extension_ws_task, spawn a tokio::spawn task that loops every 20s and sends {type:"ping", id:<uuid>} over the WS. Stops when the WS connection closes. Pong responses from the extension are received in the existing WS receiver loop and silently ignored. This keeps the MV3 service worker alive past Chrome's 30s idle deadline.

  2. Task 4.2: Extension pong handler

    Add if (msg.type === "ping") ws.send(JSON.stringify({type:"pong", id:msg.id})); before the call_tool/discover_tools handlers in the ws.onmessage callback. Matches the @mcp-b/transports BACKGROUND_MESSAGE_TYPES naming convention.

5. Zod Validation

  1. Task 5.1: Add zod dependency

    Add "zod": "^3" to webmcp-extension/package.json devDependencies or dependencies.

  2. Task 5.2: Zod-validate tool args before forwarding to page

    In the call_tool handler, define a Zod schema for args and validate before calling document.modelContext.executeTool. Invalid args return a tool_error envelope with the validation error message, never forwarded to the page. This satisfies AGENTS.md:65 mandate "Zod validation for extension-side tool arguments before forwarding to the page."

6. Cleanup

  1. Task 6.1: cargo fmt

    Run cargo fmt to resolve the formatting drift noted in the Sprint 1 audit (import ordering, line-wrapping). After this, cargo fmt --check passes clean.

  2. Task 6.2: Drop dead ExtensionRequest.id + cargo clippy clean

    Remove the id field from ExtensionRequest (it was stored but never read in the WS send loop). Closes the dead_code clippy warning so cargo clippy --all-targets -- -D warnings passes clean.

  3. Task 6.3: Simplify fixture page and drop background tab tracker hack

    Remove the setTimeout polling loop from fixtures/hello.html — with MAIN-world+document_start polyfill injection, the tool registration can call registerTool() at top level with no waiting. Remove the fixtureTabId/getTargetTabs Sprint-1 workaround from background.ts; keep a simple chrome.tabs.query({}) fallback (reliable now that the SW won't go idle mid-call).

7. Verify & Audit

  1. Task 7.1: Run all 13 verification gates

    Gates: (1) cargo fmt --check clean, (2) cargo clippy -D warnings clean, (3) cargo build --release clean, (4) npx tsc --noEmit clean, (5) npm run build clean, (6) Docker rebuild succeeds, (7) container boots all 4 ports, (8) determinism gate — 10 sequential execute_tool calls return greeting, not null, (9) tools/list returns 2 server tools with schemars schemas, (10) discover_tools returns page-registered hello, (11) structured logging ≥10 events, (12) stdout capture empty, (13) SW alive >60s with ping/pong every 20s.

  2. Task 7.2: Prove race elimination with determinism loop

    In a freshly-booted container, wait only 3 seconds (not the previous 12s), then issue tools/call execute_tool {"name":"OpenCode"} 10 times in a tight loop with no sleeps. All 10 responses MUST contain {"greeting":"Hello, OpenCode! from WebMCP sandbox"} — not null, not "tool not found", not an error. This gate fails the sprint if any call returns the wrong result. It demonstrates the page-script-timing problem is gone at the protocol level.

  3. Task 7.3: Produce docs/audits/sprint-2-verification.html

Flip task checkboxes via mark-tasks-complete.mjs. Update epic-overview.html phase-structure table to include Sprint 2.

Correction (v0.1 Sprint 1)

Sprint 2's claim that the MCP server uses rmcp::StreamableHttpService (Task 1.2) was aspirational at the time of writing. The actual transport shipped in v0.0 was a hand-rolled legacy SSE handler (GET /sse emits one endpoint:/message event then idles; POST /message replies with stateless JSON-RPC responses). The canonical rmcp::StreamableHttpService (with proper Mcp-Session-Id lifecycle and SSE-streamed results) was implemented in v0.1 Sprint 1. See Docs/epics/v0.1-exercise-hardening/phase-0/sprint-1-truth-and-transport/sprint.html for the resolution.
