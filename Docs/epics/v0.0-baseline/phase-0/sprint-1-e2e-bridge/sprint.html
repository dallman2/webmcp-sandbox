Sprint 1: End-to-End Bridge

Phase: 0 Sprint: 1 Depends on: Sprint 0

Objective

Promote the scaffold to a functional minimal baseline: replace the stub Rust server with an rmcp-backed SSE server on :3000, add the missing :8765 WebSocket relay to the browser extension, expose typed #[rmcp::tool] definitions whose JSON schemas are derived via schemars, wire full structured logging, and prove the discover→call→respond loop with an end-to-end Docker smoke test against a fixed fixtures/ page using only document.modelContext — never raw JavaScript.

Rationale & Context

Sprint 0 confirmed the repo boots but cannot function: the extension's background.ts tries ws://127.0.0.1:8765 and gets ECONNREFUSED; the SSE endpoint emits one hard-coded event; no rmcp dependency exists; no tool definitions; no schemars schema derivation; the --transport arg is parsed but unused; and structured logging fields (correlation_id/tool_name/ws_session_id) mandated by AGENTS.md are absent. Two additional critical blockers were discovered during Sprint 0 execution: (a) Chromium crashes on boot because the Ubuntu 22.04 chromium-browser package is a snap wrapper with no snapd available — CDP :9222 never opens and the extension never loads; (b) rustfmt and clippy are not installed in the rust-builder Docker stage, so lint audits cannot run. This sprint also captures the orphaned Sprint 0 mandate DESIGN.md and the epic-overview phase-structure table drift. All architectural choices are already made — the polyfill, the wire envelopes, the extension side all exist. This sprint is pure server-side realization plus Docker toolchain fixes plus a verification fixture; it deliberately reuses axum's ws feature rather than adding tokio-tungstenite, and pins the wire protocol to match background.ts exactly (call_tool/discover_tools/tool_result/tool_error/discover_result).

Execution Plan

1. Rust MCP Core

  1. Task 1.1: Add rmcp and schemars Cargo dependencies

    Add rmcp = { features = ["server","transport-io","transport-sse-server"] } and schemars to mcp-server/Cargo.toml per AGENTS.md mandates. No hand-written tool schemas — schemars derivation is the only path.

  2. Task 1.2: Create src/tools.rs with #[rmcp::tool] definitions

    Define typed tools in mcp-server/src/main.rs using the McpHandler impl block — at minimum discover_tools and execute_tool (the relay shell) — annotated with #[rmcp::tool] so rmcp auto-generates JSON schemas from the structs via schemars. Never hand-write schemas. Note: tools live in main.rs because only 2 tools exist; AGENTS.md mandates src/tools/ once >3 tools.

  3. Task 1.3: Replace stub SSE handler with rmcp SSE server on :3000

    Remove the one-shot stream::iter SSE stub (mcp-server/src/main.rs) and wire the rmcp SSE transport on :3000. The agent-facing surface now serves the MCP protocol rather than a hard-coded notifications/initialized event.

2. WebSocket Relay

  1. Task 2.1: Add :8765 WS listener reusing axum ws feature

    Enable axum's ws feature and add a second router binding on 0.0.0.0:8765 accepting upgrade requests. No new WS crate (tokio-tungstenite explicitly rejected). This closes the gap Sprint 0 task 1.2 documented.

  2. Task 2.2: Forward and correlate tool envelopes to the extension

    Implement the relay: on inbound agent tool call, forward a {type:"call_tool"|"discover_tools", id, ...} envelope over the extension's WS; on inbound {type:"tool_result"|"tool_error"|"discover_result", id, ...}, correlate by id and republish to the SSE client awaiting that call. Wire protocol matches background.ts:16-70 exactly — no protocol drift introduced on either side.

3. Structured Logging

  1. Task 3.1: Add correlation_id / tool_name / ws_session_id tracing fields

    Per the AGENTS.md observability mandate, add correlation_id (= JSON-RPC request id), tool_name, and ws_session_id as tracing span/fields on every log event that touches a tool call. Stderr-only output is already correct (mcp-server/src/main.rs); verify no println! regressions are introduced.

  2. Task 3.2: Stdout-clean smoke test (stream-corruption guard)

    Add a verifiable check that during MCP traffic the process writes nothing to stdout (the mandated guard against corrupting the JSON-RPC stream, especially under the stdio transport). Capture stdout of a tool-call burst; assert empty.

4. Docker Toolchain Fixes & E2E Verify

  1. Task 4.0: Fix Chromium runtime — replace snap-wrapped chromium-browser

    Gates all E2E testing. Sprint 0's #1 critical finding: Ubuntu 22.04 chromium-browser package is a snap wrapper shim; snap is not available in the Docker container; Chromium exits immediately on boot. Fix by replacing the runtime base image (switch to Debian bookworm which has a real chromium package) or downloading a direct Chromium .deb from Google. After this task, docker exec <ctr> curl http://127.0.0.1:9222/json/version must return a valid DevTools response. All group-4 tasks depend on this.

  2. Task 4.1: Add fixtures/hello.html (polyfill-registering test page)

    Create fixtures/hello.html that loads @mcp-b/webmcp-polyfill and calls document.modelContext.registerTool() with a minimal no-op tool. Kept out of the Docker runtime stage so the production image is unchanged; served inside the container via a one-line static server only during the smoke test.

  3. Task 4.2: Rebuild image and run end-to-end smoke loop

    Prerequisite: task 4.0 (Chromium must boot). Rebuild docker build -f docker/Dockerfile -t webmcp-sandbox:latest ., run it mounting the fixture, and exercise the full loop: agent→SSE:3000→rmcp→WS:8765→extension→document.modelContext.executeTool()→page tool→back. Confirm the typed JSON payload makes a complete round trip with a schema-validated result.

  4. Task 4.3: Assert all mandates in the E2E smoke

    The smoke must assert: no stdout pollution (task 3.2); every tool-call log line carries correlation_id/tool_name/ws_session_id; tool inputs are schema-validated through schemars-derived schemas; no eval()/page.evaluate() anywhere in the relay path — only document.modelContext.executeTool()/.getTools().

5. Hardening

  1. Task 5.1: Implement --transport stdio path

    The --transport clap arg is parsed but unused. Implement the stdio branch using the transport-io rmcp feature so agents can connect over stdio as well as SSE. This is what makes the stdout-clean mandate (task 3.2) load-bearing rather than cosmetic.

  2. Task 5.2: Fix README/AGENTS code drift (tools.rs truthfulness)

    Update mcp-server/README.md so its description of src/tools.rs and the tool surface matches what task 1.2 actually created. Closes the drift flagged in Sprint 0 task 1.3 — after this, mcp-server/README.md:15 is no longer fictional.

  3. Task 5.3: Install rustfmt + clippy in rust-builder Docker stage

    Sprint 0 task 3.1 found that cargo fmt --check and cargo clippy both error out because the Rust toolchain in the rust:1-bookworm builder image lacks these components. Add rustup component add rustfmt clippy to the rust-builder stage so future lint audits actually execute.

  4. Task 5.4: Sync epic-overview.html phase-structure table and acceptance checkboxes

    The epic overview's phase-structure table still shows only Sprint 0 (rowspan=1, status "Planned") — Sprint 1 is not listed. Manually add the Sprint 1 row. Also flip the acceptance checkboxes that Sprint 0 satisfied: "Architecture doc exists", "Audit doc exists", "Sprint 0 tasks populated".

6. Doc Closure

  1. Task 6.1: Produce e2e-bridge-verification audit doc

    Create Docs/audits/e2e-bridge-verification.html from Docs/templates/audit.html capturing Sprint 1's E2E smoke results: the round-trip payload trace, the mandate-assertion outcomes from task 4.3, and any residual gaps. Separate from baseline-quality.html to keep the pre-implementation baseline and post-implementation verification chronologically distinct.

  2. Task 6.2: Regenerate indices and validate docs

Fix any violations until clean; flip task checkboxes as work lands via mark-tasks-complete.mjs.

  3. Task 6.3: Create DESIGN.md design tokens

    AGENTS.md:160 lists DESIGN.md as a reference file "(needs creation in Sprint 0)" but no Sprint 0 task captured it. Create the file with design tokens relevant to webmcp-sandbox: color palette, typography, spacing scale. Resolves the remaining AGENTS.md drift from Sprint 0.
