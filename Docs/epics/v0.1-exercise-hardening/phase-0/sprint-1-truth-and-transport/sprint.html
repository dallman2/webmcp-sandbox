Sprint 1: Truth Reconciliation & Canonical Transport

Phase: 0 Sprint: 1 Depends on: —

Objective

Close the SSE-transport doc-vs-code divergence by replacing the hand-rolled legacy sse_handler and message_handler with rmcp::StreamableHttpService (the canonical transport the v0.0 Sprint 2 docs already claimed), add a tokio::time::timeout to the extension relay to prevent hanging tool calls, investigate and fix the polyfill executeTool-returns-null bug documented in sprint-2-verification §Residual, and reconcile all v0.0 documentation drift.

Rationale & Context

v0.0 Sprint 2's docs claim the MCP server uses rmcp::StreamableHttpService and that the hand-rolled message_handler (~110 lines of JSON-RPC string dispatch) was deleted. Reality: mcp-server/src/main.rs:260-382 still contains both the legacy sse_handler and message_handler; Cargo.toml enables feature transport-streamable-http-server but it's unused. The hand-rolled transport is not interoperable with real MCP Streamable-HTTP clients (MCP inspector, @modelcontextprotocol/sdk), which would defeat Goal #2 ("prove WebMCP works against localhost:8180"). Meanwhile, the polyfill executeTool returns null for registered tools (sprint-2-verification §Residual), so even the self-smoke fixture can't produce a real execute result. The relay_to_extension oneshot channel has no timeout — if the extension dies mid-call, the MCP request hangs forever. All of these block progress on Sprints 2 and 3.

Execution Plan

1. Verify rmcp API (Spike)

  1. Task 1.1: Resolve rmcp version and confirm StreamableHttpService API

    Check Cargo.lock for the resolved rmcp 1.x.y version. Read the rmcp source (cargo doc or docs.rs) for StreamableHttpService::new() constructor and StreamableHttpServerConfig signature. Verify it can wrap an existing McpHandler that already has #[tool_router(server_handler)]. If the API is incompatible or unavailable, fall back to Option B (keep legacy transport, file v0.1.1 task, downgrade Goal #2 proof). Timebox: 4 hours. Outcome: confirmed API signature + usage example.

2. Investigate & Fix Polyfill execute_tool Null Bug

  1. Task 2.1: Reproduce null bug with CDP inspection

    Load the fixture page in Chrome with the extension attached. Via CDP (port :9222), evaluate document.modelContext.getTools() to confirm the hello tool is registered. Call document.modelContext.executeTool({name:"hello"}, JSON.stringify({name:"OpenCode"})). Observe the return value. Compare against the polyfill's expected return shape (@mcp-b/webmcp-polyfill type definitions). Document the actual vs expected API mismatch. See sprint-2-verification §Residual for the audit's hypothesis: the polyfill may expect the execute callback to receive deserialized args and return a synchronous value, or may require a specific async binding pattern.

  2. Task 2.2: Fix the null bug (fixture or calling path)

    Based on Task 2.1 findings: (a) if the fixture's execute function signature is wrong, update fixtures/hello.html to match the polyfill's expected contract; (b) if background.ts:callPageTool passes args incorrectly (e.g., double-JSON.stringify), fix the calling path; (c) if the polyfill itself has a bug, document the root cause and create a patch or workaround. Goal: tools/call execute_tool {tool_name:"hello", args:{name:"OpenCode"}} returns {"greeting":"Hello, OpenCode! from WebMCP sandbox"} — not null. If upstream library issue beyond 1-day fix, document with clear patch path and carry to Sprint 2.

3. Implement Canonical StreamableHttpService Transport

  1. Task 3.1: Wire StreamableHttpService in place of legacy handlers

    Import rmcp::transport::streamable_http_server::{StreamableHttpService, StreamableHttpServerConfig}. Create a StreamableHttpService wrapping McpHandler { state: state.clone() }. Mount on the :3000 axum Router at /mcp using Router::route_service("/mcp", service). Delete sse_handler (lines 260-271), message_handler (lines 273-382), and remove the old /sse + /message routes from mcp_app (lines 403-413). The ws_app Router (lines 415-424) and AppState (lines 24-98) are unchanged — they are transport-agnostic. After this, tools/list and tools/call are handled by rmcp's built-in ServerHandler::handle_request — no more raw JSON-RPC string matching.

  2. Task 3.2: Pin rmcp version and verify feature flags

    Commit Cargo.lock to the repo (ensure it exists and is tracked). Verify transport-streamable-http-server feature is the only transport feature enabled (drop any transport-io or transport-sse-server if present — neither currently exists in Cargo.toml). Pin rmcp to the resolved 1.x.y major.minor.patch to prevent silent API shifts.

4. Add Relay Timeout

  1. Task 4.1: Wrap relay_to_extension oneshot with tokio::time::timeout

    In AppState::relay_to_extension (lines 89-96), replace bare rx.await with tokio::time::timeout(Duration::from_secs(30), rx).await. On Elapsed, remove the pending entry from self.pending and return Err("extension relay timeout (30s)"). Import std::time::Duration (already imported via tokio if using full feature). This prevents hanging MCP requests when the extension disconnects mid-call — a real interop bug.
