Sprint 3: Exercise Harness & Single-Command Portability

Phase: 0 Sprint: 3 Depends on: Sprint 2

Objective

Prove the sandbox works end-to-end by exercising a real MCP client (@modelcontextprotocol/sdk) against both the fixture page and the user's localhost:8180 WebMCP site, with automated assertions and clear pass/fail output. Deliver single-command portability: git clone && make run produces a working sandbox.

Rationale & Context

Sprint 1 provides a canonical Streamable-HTTP MCP server. Sprint 2 provides a correctly-booting container that can reach a host-side target. This sprint builds the proof layer: an automated exercise script that an MCP client follows to validate the entire relay chain (agent → MCP server → WS → extension → page → tool execution → result back). The exercise is the concrete deliverable for Goal #2 ("prove this thing works against localhost:8180"). The Makefile + README rewrite satisfy Goal #3 ("single command after clone to build the docker container"). An optional GHCR publish workflow is committed as workflow_dispatch-only (opt-in stretch).

Scope

In-Scope

  • MCP client smoke harness using @modelcontextprotocol/sdk StreamableHTTPClientTransport
  • Exercise flow: initialize → tools/list (assert ≥2) → discover_tools (assert non-empty) → execute_tool (assert real result, not null)
  • Dual-target support: --target fixture vs --target host with different assertion profiles
  • Fixture assertions: discover ≥1 tool (named "hello"), execute returns "Hello, OpenCode! from WebMCP sandbox"
  • Host (:8180) assertions: discover non-empty (user's tools), execute returns non-null non-error result
  • Transcript output to exercises/output/<timestamp>.json
  • exercises/expected.json for configurable assertion parameters
  • exercises/playbook.md mirroring AGENTS.md [CRITICAL PROTOCOL] workflow loop
  • Makefile with targets: build, run, run-fixture, run-host, smoke, smoke-fixture, smoke-host, publish, clean
  • OS-detection in run-host: macOS → host.docker.internal; Linux → --network=host
  • docker-compose.yml (optional) with extra_hosts / network_mode and environment block
  • README full rewrite: lead with make run, document env vars, link exercise section
  • opt-in GHCR publish workflow (.github/workflows/docker-publish.yml, workflow_dispatch only, amd64)
  • Create Docs/audits/v0.1-exercise-verification.html audit documenting the exercise results
  • Check v0.0 epic acceptance boxes if genuinely met by v0.1 close-out

Out-of-Scope (Deferred)

  • GHCR publish as automated gate on tag push (opt-in workflow_dispatch only)
  • arm64 / Apple Silicon native image
  • Multi-agent exercise scenarios
  • CI pipeline beyond the opt-in GHCR workflow

Execution Plan

1. Exercise Harness

  1. Task 1.1: Create exercises/ package with MCP SDK client

    Create exercises/ directory with package.json depending on @modelcontextprotocol/sdk. Implement smoke.mjs: connect via StreamableHTTPClientTransport to http://localhost:3000/mcp, run the playbook sequence, assert each step, write transcript to exercises/output/<ts>.json, exit 0 on pass / non-zero on failure with clear diff output.

  2. Task 1.2: Implement dual-target assertion profiles

    Create exercises/expected.json: {"fixture":{"discoverCountMin":1,"discoverContains":"hello","executeResultContains":"Hello, OpenCode"},"host":{"discoverCountMin":1,"executeResultContains":null}}. In smoke.mjs, read the appropriate profile based on --target fixture|host flag. Fixture: strict assertions for the known "hello" tool. Host: only assert non-empty discover and non-null/non-error execute result (user's site contract is unknown but confirmed WebMCP-enabled per user).

  3. Task 1.3: Create exercises/playbook.md documenting the exercise flow

    Document the MCP client sequence mirroring AGENTS.md [CRITICAL PROTOCOL]: (1) tools/list → verify server tools, (2) tools/call discover_tools → assess page tools, (3) tools/call execute_tool → execute a SINGLE tool with typed JSON args, (4) wait for response. Include expected output examples for both fixture and host targets.

  4. Task 1.4: Create exercises/run-exercise.sh driver script

    Shell script that (a) boots the container with the right env/network flags for the target, (b) waits for readiness via /sandbox-config poll, (c) runs node exercises/smoke.mjs --target <fixture|host>, (d) prints pass/fail with color output and transcript path.

2. Single-Command Portability

  1. Task 2.1: Create Makefile with build, run, smoke targets

    Create Makefile at repo root with targets: build (docker build), run (build → run with fixture default), run-fixture (build → run fixture self-test), run-host (build → run with host :8180 env), smoke (runs exercise against running container), smoke-fixture (run + smoke against fixture), smoke-host (run + smoke against host), publish (echo instructions for GHCR workflow_dispatch), clean (docker rm). Detect OS to set correct docker flags (macOS: --add-host=host.docker.internal:host-gateway; Linux: optional --network=host). make run is the canonical single command satisfying Goal #3.

  2. Task 2.2: Create docker-compose.yml

    Create docker-compose.yml with services.sandbox: build from docker/Dockerfile, ports 3000:3000 + 5900:5900, environment block for SANDBOX_TARGET_URL + SANDBOX_TARGET_HOST, extra_hosts: ["host.docker.internal:host-gateway"] for macOS, network_mode: host for Linux (comment-switch). Users can docker compose up as an alternative to make run.

  3. Task 2.3: Create opt-in GHCR publish workflow

    Create .github/workflows/docker-publish.yml: workflow_dispatch trigger (manual only), build multi-stage Docker image for linux/amd64, push to ghcr.io/<owner>/webmcp-sandbox:latest. Uses docker/build-push-action. Comment out linux/arm64 platform (deferred). Document in README "Pull & Run" section.

3. README Rewrite & Doc Audit

  1. Task 3.1: Full README rewrite

    Rewrite README.md: lead with make run as the canonical Quick Start (≤5 lines from fresh clone). Add sections: "Pull & Run" (GHCR opt-in), "Target Your Site" (SANDBOX_TARGET_URL + networking table macOS vs Linux), "Exercise" (make smoke + playbook), "Advanced" (docker-compose, manual docker run with all flags), "Development" (cargo run, npm run dev). Remove stale --transport sse --port 3000. Document all env vars.

  2. Task 3.2: Create v0.1 exercise verification audit

    Scaffold Docs/audits/v0.1-exercise-verification.html from Docs/templates/audit.html. Document: Sprints 1-3 summary, fixture exercise transcript (gate results), host exercise transcript (:8180 tools discovered/executed), container health metrics, known-issue block (polyfill null bug status), recommendations for future work.

  3. Task 3.3: Check v0.0 epic acceptance boxes if genuinely met

    Review the 2 unchecked v0.0 acceptance boxes: "serve.mjs serves the full Docs/ tree" and "validate-docs.mjs passes with zero violations." If both are genuinely met by v0.1 close-out (after Sprint 3 doc regeneration), check the boxes. Otherwise leave them unchecked with a note referencing v0.1 status.

4. Final Validation

  1. Task 4.1: Run all verification gates and close out

    Gates: (1) make build && make smoke-fixture: green transcript, discover returns hello, execute returns greeting, (2) make smoke-host: green transcript against user's :8180, (3) docker compose up reaches parity with make run-fixture, (4) node Docs/scripts/validate-docs.mjs → 0 violations, (5) node Docs/scripts/check-doc-drift.mjs → 0 drifts, (6) node Docs/scripts/query-open-tasks.mjs → 0 open in v0.1. All gates pass → mark sprint + epic complete.
