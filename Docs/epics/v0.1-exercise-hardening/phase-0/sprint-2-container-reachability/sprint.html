Sprint 2: Container Correctness & Host Reachability

Phase: 0 Sprint: 2 Depends on: Sprint 1

Objective

Make docker build && docker run actually start a working sandbox: serve the fixture page correctly, reach a host-side localhost:8180 WebMCP site cross-platform (macOS and Linux), make the extension target the right tab by URL/origin, and introduce runtime configuration (/sandbox-config endpoint) to replace hardcoded wiring.

Rationale & Context

The Docker container has four runtime failures: (1) docker/Dockerfile never copies fixtures/ into the image, so start.sh's python3 -m http.server --directory /fixtures serves nothing; (2) python3 is not explicitly installed; (3) the Chromium launch URL is hardcoded to localhost:8080/hello.html with no mechanism to target a different site; (4) container localhost ≠ host localhost on macOS (Docker VM), so the user's localhost:8180 is unreachable. Additionally, the extension has hardcoded ws://127.0.0.1:8765 in background.ts:4 and uses naive chrome.tabs.query({}) → tabs[0] for tab selection at lines 44 and 74 — so it hits the first tab regardless of whether that tab has WebMCP tools.

Scope

In-Scope

  • Serve fixtures from the Rust MCP server via tower-http::services::ServeDir at GET /fixtures/* on :3000 — drop python3 -m http.server entirely
  • Copy fixtures/ into the Docker runtime image
  • Add GET /sandbox-config endpoint returning {"wsUrl","targetUrl","targetOrigin"} for runtime configuration of the extension
  • Replace hardcoded WS_URL in background.ts:4 with /sandbox-config fetch (with literal fallback)
  • Replace tabs[0] tab selection with URL/origin-filtered chrome.tabs.query({url}) with fallback chain
  • Add page_ready message from content.ts to background SW for tab registration
  • Parameterize Chromium launch URL via SANDBOX_TARGET_URL env var (default: fixture self-test)
  • Implement host-reachability via SANDBOX_TARGET_HOST env var with localhost/127.0.0.1 → host.docker.internal rewriting
  • Add health-readiness gate before Chromium launches (poll /sandbox-config)
  • Replace bare wait with trap 'kill 0' and background PID tracking in start.sh
  • Document cross-platform networking: macOS --add-host=host.docker.internal:host-gateway vs Linux --network=host

Out-of-Scope

  • Exercise harness / smoke scripts
  • Makefile / docker-compose / GHCR
  • Multi-session AppState
  • Additional tools beyond discover_tools / execute_tool

Execution Plan

1. Fixture Serving (Rust ServeDir)

  1. Task 1.1: Add tower-http ServeDir for fixture serving

    Add tower-http = { version = "0.4", features = ["fs"] } to Cargo.toml dependencies. In main.rs, add .nest_service("/fixtures", ServeDir::new("./fixtures")) to the :3000 axum Router (alongside the existing StreamableHttpService and WS routes). Drop python3 -m http.server 8080 --directory /fixtures from docker/start.sh entirely.

  2. Task 1.2: Copy fixtures/ into Docker image

    Add COPY fixtures/ ./fixtures/ to the runtime stage of docker/Dockerfile (after the server/extension COPY lines). .dockerignore does not exclude fixtures/ — no change needed there. Verify the Rust server's working directory is /app where ./fixtures resolves correctly.

2. Runtime Configuration Endpoint

  1. Task 2.1: Add GET /sandbox-config endpoint

    Add a new axum route .route("/sandbox-config", get(sandbox_config_handler)) to the :3000 Router returning JSON: {"wsUrl":"ws://127.0.0.1:8765","targetUrl":"","targetOrigin":""}. Read SANDBOX_TARGET_URL from env (default http://localhost:3000/fixtures/hello.html). Derive targetOrigin from the URL. Serve as application/json.

3. Extension Runtime Wiring

  1. Task 3.1: Fetch WS URL and target origin from /sandbox-config

    Replace hardcoded const WS_URL = "ws://127.0.0.1:8765" (line 4) with a startup fetch("http://127.0.0.1:3000/sandbox-config") → parse {wsUrl, targetOrigin}. Store as module-level mutable state. Fall back to the literal constant if fetch fails (MV3 service worker may suspend before fetch completes). Use the wsUrl from config for the WebSocket connection.

  2. Task 3.2: Replace tabs[0] with URL/origin-filtered tab selection

    In both the call_tool handler (line 44) and discover_tools handler (line 74), replace const tab = tabs[0] with URL-filtered selection: chrome.tabs.query({url: targetOrigin + "/*"}); if empty, filter all {url: ["http://*/*", "https://*/*"]} tabs by new URL(tab.url).origin === targetOrigin; fallback chain: url-matched → origin-matched → active → first tab. If no tab matches, send a structured error back over WS: {"type":"tool_error","id":msg.id,"error":"no tab matching target origin "}.

  3. Task 3.3: Send page_ready message from content script

    After polyfill initialization in content.ts, send a chrome.runtime.sendMessage({type:"page_ready", url: window.location.href}) to the background service worker. The background SW stores this as the live target tab URL, complementing the url-filter approach.

4. start.sh Hardening

  1. Task 4.1: Parameterize Chromium launch URL and add host rewriting

    Add env var reads at top of start.sh: SANDBOX_TARGET_URL="${SANDBOX_TARGET_URL:-http://localhost:3000/fixtures/hello.html}", SANDBOX_TARGET_HOST="${SANDBOX_TARGET_HOST:-}". If SANDBOX_TARGET_HOST is set and SANDBOX_TARGET_URL contains localhost or 127.0.0.1, rewrite to the target host via Bash parameter expansion. Pass $SANDBOX_TARGET_URL as Chromium's opening URL (replacing hardcoded http://localhost:8080/hello.html on line 26).

  2. Task 4.2: Add health-readiness gate before Chromium launch

    Replace sleep 2 with an active readiness loop: until curl -sf http://127.0.0.1:3000/sandbox-config >/dev/null 2>&1; do sleep 0.5; done (timeout ~30s). Only launch Chromium after the MCP server responds. Log "server ready" on success, fail loudly on timeout.

  3. Task 4.3: Robust background process handling

    Replace bare & + wait with: trap 'kill 0' EXIT INT TERM at the top (after set -e), track PIDs explicitly (PID_XVFB=$!), and wait -n or individual wait per process. On any child exit, the trap kills the process group. Remove python3 -m http.server entirely (replaced by ServeDir in Task 1.1).

5. Verify & Validate

  1. Task 5.1: Run all container verification gates

    Gates: (1) docker build succeeds with no python3 in apt list, (2) docker run boots: curl :3000/sandbox-config returns JSON, curl :3000/fixtures/hello.html returns fixture, (3) VNC (:5900) shows Chromium on fixture with "Tool hello registered" status, (4) macOS: --add-host=host.docker.internal:host-gateway -e SANDBOX_TARGET_HOST=host.docker.internal -e SANDBOX_TARGET_URL=http://localhost:8180/ reaches host :8180, (5) Linux: --network=host -e SANDBOX_TARGET_URL=http://localhost:8180/ reaches host :8180, (6) kill container → all PIDs die (no orphans), (7) extension reconnects on WS drop (existing setTimeout(connect,2000) + new /sandbox-config fallback).

Verification & Acceptance Criteria

  • docker build succeeds; no python3 in apt list; COPY fixtures/ present docker build -f docker/Dockerfile -t webmcp-sandbox .
  • Container boots: /sandbox-config returns JSON; /fixtures/hello.html serves fixture docker run -e SANDBOX_TARGET_URL=http://localhost:3000/fixtures/hello.html -p 3000:3000 -p 5900:5900 webmcp-sandbox
  • VNC shows Chromium on fixture page with "Tool hello registered" Connect VNC client to localhost:5900
  • macOS: container reaches host :8180 via host.docker.internal docker run --add-host=host.docker.internal:host-gateway -e SANDBOX_TARGET_HOST=host.docker.internal -e SANDBOX_TARGET_URL=http://localhost:8180/ -p 3000:3000 -p 5900:5900 webmcp-sandbox
  • Linux: container reaches host :8180 via --network=host docker run --network=host -e SANDBOX_TARGET_URL=http://localhost:8180/ webmcp-sandbox
  • Background processes terminate cleanly on container kill (no orphans) docker kill && docker ps -a | grep
  • Extension selects tab matching target URL origin (not always tabs[0]) Open fixture + separate target tab; verify tools/call hits target tab
