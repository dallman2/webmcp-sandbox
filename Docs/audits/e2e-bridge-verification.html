Post-Sprint-1 Gaps

  1. Polyfill tool registration race: the fixture page registers tools via setTimeout polling for document.modelContext, but relay calls may execute before the poll fires. Resolution: use synchronous detection or MutationObserver in the fixture, or pre-register tools in the content script.
  2. #[tool_router] macro compatibility: rmcp v1.8.0's #[tool_router]+#[tool_handler] macros failed with multi-parameter tool functions. Root cause unclear — likely a trait bound issue with the tool return type. Workaround: direct schema_for!() + manual dispatch.
  3. cargo fmt --check drift: cosmetic formatting differences found (line wrapping, import ordering). Non-blocking; fix in follow-up.
  4. cargo clippy dead_code: one warning on ExtensionRequest.id field. Addressed with #[allow(dead_code)].
  5. Stdio transport stubbed: the --transport stdio path logs a warning and exits. Full rmcp stdio support requires a ServerHandler impl that matches the generated macro API. SSE is the primary transport.
  6. Extension service worker auto-disconnect: MV3 service workers disconnect after ~30s idle, dropping the WS connection. Solved for testing by keeping relay calls within the window; future: add WS heartbeat or keep-alive.

Recommendations

Immediate

  • Fix the fixture page's tool registration race (synchronous detection or MutationObserver)
  • Run cargo fmt to resolve formatting drift; verify clippy passes clean
  • Add a keep-alive mechanism for the extension's WS connection (ping/pong between server and extension)

Strategic (future sprints)

  • Resolve rmcp #[tool_router] macro compilation issue or migrate to a future rmcp release with fixed multi-param support
  • Implement full --transport stdio using rmcp's transport-io feature
  • Add vitest unit tests for the extension; add #[test] for the relay logic
  • Add Zod validation to background.ts tool arguments per AGENTS.md:65
  • CI pipeline: automate Docker build + E2E smoke on push
