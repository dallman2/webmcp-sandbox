Audit: Sprint 2 Verification

Epic: v0.0-baseline Date: 2026-06-26 Sprint 2 post-implementation

Executive Summary

Sprint 2 closed the three architectural gaps identified after Sprint 1: (a) stdio transport was dropped and the SSE layer was refactored — the server is now HTTP-only; (b) the polyfill is injected into the page's MAIN world at document_start via manifest static content_scripts, eliminating the cross-realm deadlock that caused tools to never be visible; (c) multi-param tool inputs use the Parameters<T> wrapper so #[tool_router(server_handler)] generates correct CallToolHandler impls with schemars-derived schemas. Additionally: WebSocket keep-alive (server-initiated ping every 20s), Zod validation on extension tool arguments, cargo fmt formatting, removal of dead code, and fixture page simplification (no more polling). All three root causes are resolved. One residual issue remains: the @mcp-b/webmcp-polyfill v3.0.0 executeTool API returns null when called with a tool name string or object — discoverPageTools/getTools() confirms the tool is registered, the relay chain is proven (a hardcoded return value in callPageTool correctly traverses the full round trip), but the polyfill's own execution path returns null. This is a library compatibility issue tracked for follow-up.

Residual: Polyfill executeTool Returns Null

  • What works: the polyfill's getTools() confirms the fixture's "hello" tool is registered in a single StrictWebMCPContext Map. The relay chain (server → WS → extension → injected main-world function → polyfill → back) is proven correct by injecting a function that returns a hardcoded string — the result arrives intact through all layers.
  • What doesn't: calling document.modelContext.executeTool({name:"hello"}, argsJson) (v3.0.0 expects a tool object with .name) or navigator.modelContextTesting.executeTool("hello", argsJson) (testing shim, accepts a string name) both return null instead of the serialized result. The tool IS found (no "Tool not found" error thrown), so the lookup succeeds. The execute function IS called. But the return value from the polyfill's executeToolByName comes back null.
  • Root cause hypothesis: the @mcp-b/webmcp-polyfill v3.0.0 executeToolByName method at line ~110 serializes the raw result via JSON.stringify(K) and returns null if the result is undefined === void 0. This means K (the resolved return from the tool's execute function) is undefined. Possible causes: (a) the args parsed by the polyfill don't reach the execute function correctly — the fixture's execute receives undefined instead of {name: "OpenCode"}; (b) the execute function is called with incorrect this binding; (c) a silent TypeError inside the execute function is caught by the polyfill and converted to null. The round-trip infrastructure (content script world, relay, event flow) is not the bottleneck.
  • Resolution track: needs a separate investigation where the polyfill's internal state is logged at each step (args parsing, tool lookup, execute invocation, return value). The CDP WS endpoint is available for direct page evaluation; this pinpoints whether the issue is in the polyfill's deserialization or the tool execution itself.
  • Risk: low — the architecture around it is solid. This is a single-library API compatibility detail, not a design flaw.

Recommendations

  1. Implement canonical StreamableHttpService: the v0.0 Sprint 2 plan claims the server uses rmcp::StreamableHttpService but the actual code still runs a hand-rolled legacy SSE handler (GET /sse + POST /message). The canonical transport (with its Mcp-Session-Id lifecycle and SSE-streamed results) should be implemented in v0.1 Sprint 1 to ensure real MCP clients (MCP inspector, @modelcontextprotocol/sdk) can interoperate.
  2. Investigate polyfill executeTool null bug via CDP: the @mcp-b/webmcp-polyfill v3.0.0 executeTool returns null for registered tools (see §Residual). Use CDP (:9222) to log polyfill internal state at each step (args parsing, tool lookup, execute invocation, return value) — pinpoint whether the issue is deserialization or execution. If fixable in ≤1 day, patch the fixture or calling path. If upstream library, document root cause and patch path.
  3. Add relay timeout: AppState::relay_to_extension uses a bare oneshot with no timeout — if the extension disconnects mid-call, the MCP request hangs forever. Add tokio::time::timeout(Duration::from_secs(30), rx.await).
  4. Add missing #recommendations section: this audit file was missing the required <section id="recommendations"> per Spiderweb audit template schema. Fixed.
