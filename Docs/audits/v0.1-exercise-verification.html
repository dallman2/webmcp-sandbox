Audit: v0.1 Exercise Verification

Epic: v0.1-exercise-hardening Date: 2026-06-26

Executive Summary

This audit verifies the v0.1-exercise-hardening epic across three sprints: Sprint 1 (Truth Reconciliation & Canonical Transport) replaced the hand-rolled SSE handler with rmcp::StreamableHttpService, fixed the polyfill execute_tool null bug, and added a 30s relay timeout. Sprint 2 (Container Correctness & Host Reachability) fixed Docker runtime failures — fixture serving via tower-http ServeDir, /sandbox-config runtime config, URL/origin-filtered tab selection replacing naive tabs[0], and cross-platform host reachability. Sprint 3 (Exercise Harness & Single-Command Portability) delivers the smoke harness (exercises/smoke.mjs using @modelcontextprotocol/sdk), Makefile for single-command portability, docker-compose.yml, and the opt-in GHCR publish workflow.

The sandbox now successfully completes the full relay chain end-to-end: MCP client → Streamable HTTP :3000 → Rust server → WS :8765 → extension → document.modelContext → page tool → result back.

Strengths

  • Canonical transport: rmcp::StreamableHttpService is the standard MCP transport — interoperable with MCP inspector and @modelcontextprotocol/sdk
  • Polyfill null bug fixed: execute_tool now returns real results; fixture's execute() normalizes string args via JSON.parse
  • Relay timeout: tokio::time::timeout(30s) prevents hanging MCP requests on extension disconnect
  • Container serves fixtures natively: tower-http ServeDir eliminates the python3 http.server dependency
  • Smart tab targeting: URL/origin-filtered tab selection with fallback chain replaces fragile tabs[0]
  • Runtime configuration: /sandbox-config + SANDBOX_TARGET_URL env var make the sandbox configurable without code changes
  • Dual-target exercise: smoke.mjs validates both the fixture (strict) and user's :8180 site (flexible)
  • Single-command portability: make run works from any fresh clone

Weaknesses (Critical Risks)

WeaknessRisk LevelDescription
Single-extension session Medium AppState supports only one extension WebSocket connection. Multi-agent scenarios require concurrent sessions — deferred to future epic (D0).
Tool-per-file refactor not done Low Both tools (discover_tools, execute_tool) live in main.rs. AGENTS.md mandates src/tools/ split once >3 tools — deferred as D4 since only 2 tools exist.
No CI beyond GHCR Low Verification is manual via make smoke-fixture and make smoke-host. GHCR workflow is workflow_dispatch opt-in only. Deferred CI pipeline (D3).
amd64 only Low No arm64/Apple Silicon native image. Docker Desktop emulation sufficient for v0.1. Deferred (D2).

Recommendations

Immediate (Low Effort)

  • Validate on Linux host: the smoke-host path with --network=host should be verified on a real Linux machine
  • Publish GHCR image: trigger the opt-in workflow_dispatch to publish webmcp-sandbox:latest to ghcr.io
  • Add extension unit tests: Vitest suites covering background.ts tab selection and message routing (deferred since v0.0 Sprint 1)

Strategic (High Performance)

  • Multi-agent support: refactor AppState to manage multiple extension connections with session-scoped pending maps
  • Tool-per-file refactor: once tool catalog grows >3, split into src/tools/ per AGENTS.md conventions
  • Automated CI: add push-triggered CI that runs cargo fmt/clippy/build + extension build + docker build + smoke-fixture
  • arm64 image: add linux/arm64 platform to Docker build and GHCR workflow
